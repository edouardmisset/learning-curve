## TL;DR

A Git worktree gives each coding agent its own directory and branch while the
worktrees share one repository. That isolation makes parallel changes easier to
reason about, but it does not make conflicting work disappear. Assign one issue
and a clear file scope per agent, use a different agent or human to review each
change, then integrate and test the branches together before merging.

## Parallel work needs boundaries

One coding agent can edit a checkout safely enough. Add five agents to that same
directory and the situation changes: one switches branches while another is
writing, generated files are overwritten, and nobody can tell which process
changed the lockfile. More speed has produced less control.

The obvious alternative is a fresh clone per task. It works, but duplicates Git
objects and requires every clone to fetch and configure remotes independently.
Git worktrees offer a lighter arrangement: several working directories attached
to one repository, each with its own checked-out branch.

Worktrees are not specifically an AI feature. They are a Git primitive that
happens to provide useful boundaries for humans, scripts, and autonomous coding
tools alike.

## The mental model: branch, worktree, agent

Three terms describe three different things:

- A **branch** is a movable name pointing to a commit. It records the history of
  one stream of work.
- A **worktree** is a directory containing a checked-out revision, plus private
  state such as its `HEAD` and index.
- An **agent** is the worker operating in that directory. It may edit, test, and
  commit, but its permissions and task are defined outside Git.

The linked worktrees share the repository's object database and references.
They do not share uncommitted files or staging areas. That distinction is the
whole trick: sharing history is efficient; separating mutable working state is
safe enough for concurrent tasks.

## Create a safe workspace per issue

Start by updating remote-tracking information in the main checkout. Then create
an immutable base identifier and create every author branch from that exact
commit:

```bash
git fetch origin
BASE_COMMIT=$(git rev-parse origin/main)

git worktree add -b issue-101 ../worktrees/issue-101 "$BASE_COMMIT"
git worktree add -b issue-102 ../worktrees/issue-102 "$BASE_COMMIT"
git worktree add -b issue-103 ../worktrees/issue-103 "$BASE_COMMIT"

git worktree list
```

`origin/main` is a remote-tracking name and can move after a fetch. Resolving it
once means every branch uses the same commit even if that name changes while the
batch is running. Record `BASE_COMMIT` in the coordination notes if later steps
will run in another shell. Choose branch and directory names that identify the
issue; when a terminal says `issue-102`, there should be no detective work
required.

Run each agent with its worktree as the current directory. Its assignment should
state:

1. The issue and acceptance criteria
2. Files or subsystem it owns
3. Files it must not change without escalating
4. Required checks and tests
5. The expected commit and handoff format

This is **one-issue ownership**. An agent may discover adjacent problems, but it
should report them rather than silently expanding its branch. Small, purposeful
diffs are easier to review and less likely to collide.

<Aside type="caution">
  Do not use `--force` merely to bypass Git's warning that a branch is already
  checked out. Two workers writing to the same branch make branch ownership
  ambiguous and can move its tip underneath one another.
</Aside>

Git normally refuses to check out the same branch in multiple worktrees. Treat
that safeguard as a design hint: use one branch, one worktree, and one author for
each task. A detached worktree is useful for read-only inspection, but an author
should have a named branch so its commits have an obvious home.

## Work in parallel without pretending work is independent

Separate directories prevent filesystem races; they do not prevent semantic
conflicts. Two branches can still edit the same function or make incompatible
architectural decisions. The coordinator should therefore partition work before
starting agents.

Prefer assignments with disjoint ownership. If several articles are being
written, reserve filenames and publication order centrally. If several features
touch one package, assign shared configuration to one branch or schedule it as a
small prerequisite. Record these decisions where every worker can see them.

Each author should commit cohesive changes on its own branch:

```bash
git status --short
git diff --check
git add src/feature tests/feature.test.ts
git commit -m "Add issue 101 feature"
```

Explicit paths reduce the chance of sweeping unrelated generated files into a
commit. Before handoff, the author runs the repository's formatter, type checker,
tests, and build. A clean isolated build proves that branch works alone; it does
not yet prove that all parallel branches work together.

## Review with a fresh pair of eyes

Authoring and reviewing are different jobs. Give each branch to a reviewer who
did not write it—another agent or a human—and provide the original issue, not
just the diff. Otherwise the reviewer can verify implementation quality while
missing that the implementation solves the wrong problem beautifully.

A useful review checks four layers:

- **Intent:** Does the diff satisfy the issue and remain within scope?
- **Correctness:** Are failure paths, security boundaries, and edge cases handled?
- **Evidence:** Do relevant checks pass, and do tests fail when the behavior is
  deliberately broken?
- **Integration:** Does the branch overlap shared files or assumptions from other
  active branches?

The reviewer reports blocking and non-blocking findings. The original author
addresses them so ownership stays clear, and the reviewer performs a second pass.
For high-risk changes, inspect the branch in a detached review worktree:

```bash
CANDIDATE_101=$(git rev-parse issue-101)
git worktree add --detach ../worktrees/review-101 "$CANDIDATE_101"

# Set this only after the reviewer approves that exact checkout.
APPROVED_101=$CANDIDATE_101
```

Resolving the author's branch first gives the reviewer an immutable snapshot.
Because the review checkout is detached, it cannot accidentally become a second
writer for the author's branch. If the author adds fixes, resolve the new tip to
a new candidate, recreate the detached checkout, and review it again before
updating `APPROVED_101`. Record every approved commit ID in the coordination
notes; the branch name is not the reviewed artifact. Remove the review worktree
when finished.

## Integrate before declaring victory

Parallel branches may all pass alone and fail when combined. Create a temporary
integration branch from the same base, merge the reviewed branches, and run the
full suite there:

```bash
git worktree add -b integration/parallel-batch \
  ../worktrees/integration "$BASE_COMMIT"

cd ../worktrees/integration
# APPROVED_101, APPROVED_102, and APPROVED_103 were pinned during review.
git merge --no-edit "$APPROVED_101"
git merge --no-edit "$APPROVED_102"
git merge --no-edit "$APPROVED_103"

# Replace these with the project's real commands.
pnpm install --frozen-lockfile
pnpm check
pnpm test
pnpm build
```

An integration branch is a rehearsal, not a place to hide fixes. Never resolve a
branch name again at this stage: merge the approved commit IDs so integration
tests the exact code that reviewers saw. If a merge or test fails, change the
branch that owns the problem, pin and review its new candidate, update that
issue's approved ID, and rebuild the integration branch. This preserves the
relationship between issue, diff, review, and final commit.

Integration order can matter. A foundational refactor may need to land before a
feature built on it. Independent branches can usually be merged in any order,
but dependency edges should be written down instead of discovered through a
particularly theatrical conflict.

## Failure modes to plan for

### Overlapping files and reserved names

Two agents changing the same registry, navigation file, migration sequence, or
article number will conflict even when their main implementation is separate.
Reserve scarce names centrally and give one task ownership of shared indexes.
For unavoidable overlap, sequence the branches or expect a deliberate rebase and
fresh review.

### Lockfile churn

Package manifests and lockfiles are shared hotspots. Avoid having unrelated
branches update dependencies simultaneously. If that is unavoidable, merge the
first dependency change, rebase the next branch, regenerate its lockfile with the
project's pinned package-manager version, and rerun a frozen install. Never solve
a lockfile conflict by choosing one side blindly.

### Ports, environment files, and secrets

Worktrees isolate files, not operating-system resources. Two development servers
cannot usually bind the same port. Allocate a port per worktree, use separate
local databases or schemas when state can collide, and make process names easy
to identify.

Ignored environment files are not copied automatically when a worktree is
created. Provision them through an approved local mechanism. Do not commit
secrets to make parallel setup convenient, and do not let agents print secret
values into logs or review comments.

### Dependencies and build artifacts

Linked worktrees do not share untracked dependency directories by default. Run
the normal installation in each worktree, or use the package manager's supported
global cache. Symlinking mutable dependency trees between workers can introduce
the very races the worktrees were meant to avoid.

Give each process its own build output and cache directories when tools write
outside the worktree or use a globally configured path. Also remember that Git's
worktree documentation notes incomplete support for multiple checkouts of a
superproject that uses submodules; test that repository shape before scaling it.

### Stale or dirty worktrees

Deleting a directory manually leaves administrative metadata behind. Inspect
state first, remove clean worktrees through Git, and prune records for paths that
no longer exist:

```bash
git worktree list
git -C ../worktrees/issue-101 status --short
git worktree remove ../worktrees/issue-101
git worktree prune --dry-run
git worktree prune
```

`git worktree remove` refuses a dirty worktree unless forced. That is another
useful guardrail: confirm changes are committed and pushed before cleanup rather
than reaching immediately for `--force`. The branch remains after its worktree
is removed and can be deleted separately when normal repository policy allows.

## When worktrees are the wrong tool

Worktrees are excellent for a handful of concurrent branches that share one
trusted local environment. Separate clones, containers, or remote runners are
better when workers need incompatible toolchains, stronger security isolation,
independent Git configuration, or reproducible clean-room environments.

There is also a coordination ceiling. Ten narrowly scoped tasks may parallelize;
ten agents redesigning the same central abstraction probably will not. Worktrees
reduce checkout cost, not communication cost. If most assignments touch the same
files, sequence the work or first create an agreed foundation.

## A practical operating checklist

Before starting:

- Refresh the base and reserve branches, worktree paths, filenames, ports, and
  shared resources.
- Define one issue, one owner, acceptance criteria, and required checks per branch.
- Identify dependencies and shared-file hotspots.

Before merging:

- Require a clean, committed author worktree and an independent review.
- Verify each branch alone, then verify their combination in an integration
  worktree.
- Return integration fixes to the owning branch and review them again.

After merging:

- Stop local processes, confirm work is pushed, and remove linked worktrees with
  `git worktree remove`.
- Use `git worktree list` and a dry-run prune to find stale metadata.
- Delete branches according to the team's usual retention policy.

## Conclusion

Git worktrees turn parallel coding from a shared-directory scramble into a set
of named, inspectable lanes. The durable pattern is simple: one task, branch,
worktree, and author; a different reviewer; then an integration rehearsal. The
speed comes from concurrency, but the confidence comes from explicit ownership
and evidence at every handoff.
